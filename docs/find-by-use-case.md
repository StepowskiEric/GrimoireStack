# Find by Use Case

**"I need a skill for..."**

> Use this as your first stop. Each entry describes a concrete situation, points to the right skill, and—critically—says *why that skill over others that sound similar.*

### Debugging / Bug Fixing

| When this is happening… | Use this skill | Why this one (and not the others) |
|---|---|---|
| You have a real crash or exception — a stack trace, error log, Sentry event, or console error. You need to go from "something broke" to "exactly where and why." | [`log-trace-correlation`](../debugging/log-trace-correlation/SKILL.md) | Trace-to-source mapping is the fastest path when you already have breadcrumbs. Don't reach for `bisect` unless you have no idea what broke. |
| Tests *used* to pass and now they don't. You need to find what changed, fast, without reading every diff. | [`bisect-debugging`](../debugging/bisect-debugging/SKILL.md) | Git bisect automates search. If you know *when* it worked, this is faster than any other approach. +9.9% speed in tests. |
| The error message is cryptic or missing. No stack trace. Something is wrong but you can't tell what from the output alone. | [`debug-subagent`](../debugging/debug-subagent/SKILL.md) | Spawns a dedicated debugger that investigates *before* suggesting edits. Prevents the common failure mode of patching symptoms. |
| Your test output is 2000 lines and the actual failure is buried. The noise is hiding the signal. | [`purify-test-output`](../debugging/purify-test-output/SKILL.md) | Strips pass lines, focuses on failures. ~18.6% token reduction, and the LLM actually sees the bug instead of drowning. |
| A bug that only manifests deep in a loop, inside a callback, or in production — you can't `console.log` your way through it. | [`simulate-instrumentation`](../debugging/simulate-instrumentation/SKILL.md) | Auto-inserts prints at key points, runs the test, captures actual runtime values. Unlike manual logging, it doesn't miss the variable that matters. |
| You tried to fix it once and it didn't work. The second attempt also failed. You're guessing. | [`iterative-patch-repair`](../debugging/iterative-patch-repair/SKILL.md) | Formal loop: patch → test → capture state → refine. Augments the patch with runtime context so each iteration is smarter. +19.9% from augmentation. |
| The same bug reappears after being "fixed." You're playing whack-a-mole. | [`root-cause-analysis`](../debugging/root-cause-analysis.md) | 5 Whys + Ishikawa forces causal chain tracing. Other skills find *a* fix; this one finds *the* root cause so it stays fixed. |
| You have a complex bug and want a disciplined reproduce → isolate → fix → verify cycle instead of guessing. | [`debug-issue`](../software-development/debug-issue.md) | Graph-powered tracing through call sites — unlike `log-trace-correlation`, this works when you don't have a stack trace yet. |
| You need the full arsenal — competing hypotheses, instrumentation, output purification, and iterative repair — fused into one workflow. | [`debug-to-fix-pipeline`](../debugging/debug-to-fix-pipeline.md) | Fuses 5 research-backed skills. Use when a single skill isn't enough and you need a complete pipeline from symptom to verified fix. |
| Tests are failing and you're not sure if the test is wrong or the code is wrong. You don't want to "fix" correct code to match a bad test. | [`unit-test-debugging`](../development/unit-test-debugging/SKILL.md) | Determines source of truth before acting. Prevents the expensive failure mode of changing implementation to match bad assertions. |
| You're writing Jest tests and need a reference — matchers, mocking, async patterns, configuration. | [`jest-testing`](../development/jest-testing/SKILL.md) | Comprehensive Jest reference. Not a debugging skill — a writing-reference skill. |
| You're dropped into unfamiliar code and need to understand the relevant module *before* fixing a bug in it. | [`explore-codebase`](../software-development/explore-codebase.md) | Token-efficient graph navigation from module structure → file roles → symbols. Use this *before* `debug-subagent` if you're in unknown territory. |
| The codebase is huge (monorepo, 500+ files) and you don't even know which directory the bug lives in. | [`codebase-divide-conquer-search`](../software-development/codebase-divide-conquer-search/SKILL.md) | Summarizes hierarchically, partitions by semantic similarity, spawns parallel investigators. For when `grep -r` is hopeless. |
| Same situation, but you also want to leave a retrievable trail so the next agent (or future you) knows what was explored. | [`navigator`](../orchestration/navigator/SKILL.md) | Scout reads + Thought-Retriever stores reasoning traces. Unlike divide-conquer-search, this is designed for ongoing investigation, not one-shot find. |
| Multiple root cause candidates exist and you need to pick the right one without following every false lead. | [`occam-root-cause`](../debugging/occam-root-cause/SKILL.md) | Generates competing causal chains, selects simplest that fits all evidence, *then* falsifies it before fixing. Faster than exploring every branch. |
| Same situation, but the failure is novel — you can't guess what type of bug it is from past experience. | [`specter`](../debugging/specter/SKILL.md) | Abductive reasoning + structural code location. Unlike `occam-root-cause`, this generates hypotheses from the code structure itself, not from a predefined catalog. |
| The debugging session has gone past 20 turns, context is decaying, and the agent keeps circling. | [`long-task-survival-kit`](../execution/long-task-survival-kit/SKILL.md) | Checkpoints every 5 calls to re-ground context, verify trajectory, prune stale assumptions. Use when `trajectory-guard` alone isn't enough. |
| A crash occurs at a different location than the actual root cause — e.g., null pointer at line 50 but the real bug is at line 10. | [`time-traveling-debugger`](../debugging/time-traveling-debugger/SKILL.md) | Records deterministic trace forward, replays backward from crash to find first state divergence. Unlike other debugging skills, this traces *backward*. |
| `npm install` fails, ports are taken, disk is full, Docker won't start — the environment itself is broken. | [`environment-recovery`](../debugging/environment-recovery/SKILL.md) | Structured vitals check (disk, versions, caches, ports, permissions) + targeted repair. Use *before* blaming your code. |
| A fetch fails, the API returns 403/429/500, CORS blocks the request, or a redirect chain breaks. The code looks right but the network is wrong. | [`network-api-debugging`](../debugging/network-api-debugging/SKILL.md) | Captures actual traffic, diagnoses by status code pattern, fixes by failure type. Unlike log-trace-correlation, this captures the *wire*, not the code. |
| A bug is visible at runtime (wrong render, wrong calculation) but no existing test exercises that path. | [`minimal-reproduction`](../debugging/minimal-reproduction/SKILL.md) | Writes the smallest possible test that demonstrates the bug. Having a minimal repro turns "weird behavior" into a debuggable assertion. |
| Same situation, but you have multiple ideas for what triggers the bug and want to test the simplest trigger first. | [`occam-minimal-repro`](../debugging/occam-minimal-repro/SKILL.md) | Ranks triggers by complexity, tests simplest first. Prevents building a complex test harness when a one-liner would reproduce it. |
| The agent has tried everything — multiple approaches, none worked, and it's stuck in a loop. | [`escalation-ladder`](../debugging/escalation-ladder/SKILL.md) | 5-level protocol: self-correct → strategy change → rubber duck → scope reduction → full retreat. For when even `iterative-patch-repair` stalls. |
| A fix needs changes in 3+ files — a schema change, a service update, and a test — and you need to do it without breaking intermediate states. | [`coordinated-change`](../software-development/coordinated-change/SKILL.md) | Maps the full change set, orders dependencies, edits atomically, verifies consistency. Use when `edit` + `git add` one-by-one would leave the tree broken between edits. |

**Not recommended for typical code bugs** (empirically measured):

- `abductive-first-debugging` — Merged into `specter`. Use specter directly.
- `pdca-deming` — **-1.6% vs baseline** on real repo bugs. It's a process improvement cycle, not a debugger.
- `step-level-verification-protocol` — **0% correctness improvement** on bugs (even though +9.9% time efficiency). Good for math/logic verification, bad for debugging.

### Reasoning / Problem Solving

| When this is happening… | Use this skill | Why this one (and not the others) |
|---|---|---|
| A hard problem with no clear starting point. You know the goal but don't know the first step. | [`how-to-solve-it-state-machine`](../execution/how-to-solve-it-state-machine.md) | Forces explicit problem framing *before* any action. Unlike every other reasoning skill, this one refuses to proceed until you've articulated what kind of problem this is. |
| Multiple plausible explanations, none obviously wrong. You need to pick the one to investigate first. | [`occams-razor`](../judgment-and-routing/occams-razor/SKILL.md) | Ranks alternatives by complexity, tests simplest first. Unlike `specter`, this assumes the simplest is probably right — don't use this for novel failure modes. |
| Same situation, but the failure is novel or unfamiliar — you can't rely on "the simplest explanation wins." | [`specter`](../debugging/specter/SKILL.md) | Abductive reasoning + structural code location. Unlike Occam, this generates hypotheses from *code structure* and tests each by disconfirmation. |
| All competing explanations fit the evidence equally well. You need to break the tie. | [`occam-abduction`](../judgment-and-routing/occam-abduction/SKILL.md) | Ranks by *assumption count*, not just complexity. Use when Occam's Razor gives you ties — the explanation with fewer hidden assumptions wins. |
| You know what the code should *do* but can't find where it's *done* — the function name, file, or pattern you'd grep for doesn't exist. | [`keyword-agnostic-logic-locator`](../execution/keyword-agnostic-logic-locator/SKILL.md) | Finds code by structure: call graph, data flow, type hierarchy. Unlike grep or fffind, this works when you don't know the names. |
| Multiple strategies are possible and you need to decide which to explore without wasting time on dead ends. | [`occam-mcts`](../orchestration/occam-mcts/SKILL.md) | Occam ranks branches by complexity; MCTS explores simplest viable branches first. Unlike `monte-carlo-tree-search` alone, this biases toward simpler solutions. |
| Same situation, but you prefer a more exhaustive search that doesn't bias toward simplicity. | [`tree-of-thoughts-plus-monte-carlo-tree-search`](../orchestration/tree-of-thoughts-plus-monte-carlo-tree-search.md) | ToT branches widely before MCTS prunes. Use when you suspect the correct answer is *not* the simplest one. |
| A decision with genuine trade-offs — every option has pros and cons and you need to surface the hidden conflicts. | [`jury`](../orchestration/jury/SKILL.md) | Spawns parallel perspectives with conflicting goals. Unlike six-thinking-hats, the perspectives *argue*, exposing real conflict rather than just listing viewpoints. |
| You think you understand the problem but want to verify before acting. | [`prism`](../reasoning/prism/SKILL.md) | Confidence calibration + compression test. If you can't compress the problem to its essence, you don't understand it yet. |
| You're stuck in a loop of the same failed attempts — you need to break the cognitive fixation. | [`cross-domain-analogy-generator`](../systems-and-architecture/cross-domain-analogy-generator.md) | Forces foreign-domain analogies (biology, music, traffic). Unlike every other reasoning skill, this one deliberately avoids the problem domain. |
| You keep jumping to solutions before understanding the problem. | [`ooda-loop-state-machine`](../execution/ooda-loop-state-machine.md) | Observe → Orient → Decide → Act is an enforced cycle. Unlike `how-to-solve-it`, this is designed for *fast-changing* environments where conditions shift between actions. |
| You're over-thinking a trivial decision — spending 10 minutes on something that should take 10 seconds. | [`cognitive-friction-governor`](../execution/cognitive-friction-governor.md) | Budgets deliberation per task. When the budget is exhausted, you must decide. Unlike every other skill, this one forces speed. |
| You started down a reasoning path and it went wrong — you need to backtrack before compounding the error. | [`process-reward-model-protocol`](../execution/process-reward-model-protocol/SKILL.md) | Self-assigns process rewards per step, backtracks when cumulative reward drops. Unlike `step-level-verification`, this catches later errors too, not just the immediate next step. |
| The current problem closely resembles one you've solved before. You want to import that prior solution's structure. | [`how-to-solve-it-analogy`](../execution/how-to-solve-it-analogy.md) | Structured analogy import. Unlike `cross-domain-analogy-generator`, this imports from a *known related* domain, not a deliberately foreign one. |
| You're building a long chain of reasoning (5+ steps) where an error in step 2 invalidates everything after it. | [`step-level-verification-protocol`](../execution/step-level-verification-protocol/SKILL.md) | Verifies each step before proceeding. Better than `process-reward-model` for error *prevention*, worse for catching errors that span multiple steps. |
| The agent is confidently asserting facts that may not be true — assumptions are compounding uncorrected. | [`assumption-grounding`](../execution/assumption-grounding/SKILL.md) | States every assumption, verifies with cheapest check, logs result. Unlike `claim-verification-reasoning`, this catches assumptions *before* they become claims. |
| The agent has tried the same approach 3+ times despite it failing — it's stuck in a loop. | [`trajectory-guard`](../execution/trajectory-guard/SKILL.md) | Detects repetition loops, specification drift, and stuck trajectories. Unlike `escalation-ladder`, this detects the problem automatically without waiting for N failures. |
| The agent started answering a different question than the one asked — specification drift. | [`trajectory-guard`](../execution/trajectory-guard/SKILL.md) | Re-checks alignment with original request. Same skill as above — trajectory-guard handles both loops and drift. |
| Context is running out mid-task (10+ turns in) and the agent is forgetting earlier findings. | [`context-lifecycle-manager`](../execution/context-lifecycle-manager/SKILL.md) + [`trajectory-guard`](../execution/trajectory-guard/SKILL.md) | Budget tracking + decay + pruning + failure-spiral detection. The combo prevents the two main failure modes of long sessions: context overflow and unproductive loops. |
| The session is 20+ turns and messages from turn 5 are taking up space but are never referenced. | [`context-lifecycle-manager`](../execution/context-lifecycle-manager/SKILL.md) | Exponential decay on old messages. Prunes low-weight content before the context window overflows. Extends useful context life 2-3x. |
| You want every aspect — birth, budget tracking, decay, pruning, token optimization — in one lifecycle system. | [`context-lifecycle-manager`](../execution/context-lifecycle-manager/SKILL.md) | Full lifecycle: context is born → tracked by budget → decays → pruned → optimized. Absorbs context-budget-operator, context-rot-pruner, and token-budget-operator. |

### Process Improvement

| When this is happening… | Use this skill | Why this one (and not the others) |
|---|---|---|
| You need to improve an existing process with a measurement-anchored cycle — no guesswork. | [`pdca-deming`](../execution/pdca-deming.md) | Plan → Do → Check → Act with measurable baselines. Unlike `toyota-kata`, this requires a clear baseline metric before you start. |
| You want a scientific-thinking pattern where each experiment teaches you something, whether it works or not. | [`toyota-kata-state-machine`](../execution/toyota-kata-state-machine.md) | Target condition → experiment → reflection. Unlike PDCA, this is designed for *uncertain* outcomes where the target condition itself may shift. |
| A high-stakes multi-step procedure where skipping a step could be catastrophic. Surgery, deployment, incident response. | [`checklist-manifesto`](../execution/checklist-manifesto.md) | Checklist discipline prevents skip-ahead errors. Unlike any other process skill, this one assumes expert knowledge is *not sufficient* — you need the checklist anyway. |
| Someone said "build me X" but you're not sure what X actually is, and coding it wrong would waste a week. | [`requirement-crystallization-protocol`](../execution/requirement-crystallization-protocol/SKILL.md) | Socratic questioning → Intent Specification. Surfaces the critical unknown before code is written. Unlike `blueprint`, this focuses on *capturing intent*, not structuring ambiguity. |
| Same situation, but even before clarifying requirements — before any significant action — you need a universal preflight check. | [`pre-flight-intent-verification`](../orchestration/pre-flight-intent-verification/SKILL.md) | Three-phase gate: clarify → specify → ground. Unlike `requirement-crystallization`, this also verifies factual claims before acting. |
| You want the full combo: target-condition thinking + measurement cycles + clean module design. | [`iterative-improvement-cycle`](../execution/iterative-improvement-cycle.md) | Toyota Kata + PDCA + Philosophy of Software Design fused. Use when no single methodology covers your improvement need. |
| A feature where a single bug, security flaw, or misalignment would be catastrophic. Payment, auth, medical. | [`zero-defect-protocol`](../execution/zero-defect-protocol/SKILL.md) | 14 phases: data contracting, invariant maps, red-team critique, pre-mortem, 3x expert review. Unlike any other skill, this one refuses to ship without mathematical confidence. |
| A vague problem that keeps being discussed in circles — everyone has a different mental model of what the actual issue is. | [`blueprint`](../execution/blueprint/SKILL.md) | Codifies the problem into types and schemas. Unlike `requirement-crystallization`, this reveals contradictions in the problem *itself*, not just unclear requirements. |

### Code Review / Quality

| When this is happening… | Use this skill | Why this one (and not the others) |
|---|---|---|

| You've produced an answer and want to catch errors *before* presenting it — a self-verification pipeline. | [`self-verify-pipeline`](../output-quality/self-verify-pipeline/SKILL.md) | Escalating: internal critique → claim decomposition → tool verification → external sources. Unlike a standard pre-mortem, this traces cascading consequences for each failure. |
| You need a security review covering authentication, secrets, input handling, and trust boundaries. | [`security-review-protocol`](../systems-and-architecture/security-review-protocol/SKILL.md) | STRIDE + Unsafe Control Actions + Vibe Coding Security. Three lenses fused — broader than any single security framework. |
| You need to refactor, but safely — test first, then change in small steps with verification at each step. | [`refactoring-state-machine`](../execution/refactoring-state-machine.md) | Structured refactoring with anti-loop protection. Unlike `legacy-rescue`, this assumes you *have* tests to verify against. |
| You want to follow pragmatic programming principles — reversible decisions, DRY but not premature, root-cause fixes. | [`pragmatic-programmer-state-machine`](../execution/pragmatic-programmer-state-machine.md) | Core software craftsmanship principles as a state machine. Use as a background reference during any coding task. |
| You're about to execute a task and want a quick thoroughness check — did I miss anything obvious? | [`thoroughness-check-etto`](../judgment-and-routing/thoroughness-check-etto.md) | Systematic completeness check. Lighter than `thoroughness-check-etto-state-machine` — use for everyday tasks. |
| Same situation, but the task is high-stakes and skipping a step could be costly — you want an enforced protocol. | [`thoroughness-check-etto-state-machine`](../judgment-and-routing/thoroughness-check-etto-state-machine.md) | Enforced state machine — cannot proceed without completing each gate. Use the same as above but stricter. |
| You have a decision to make with multiple options and want to test it against alternatives before committing. | [`counterfactual-policy-testing`](../judgment-and-routing/counterfactual-policy-testing.md) | Tests against null, opposite, and partial alternatives. Unlike `jury`, this doesn't just argue — it simulates outcomes for each counterfactual. |
| Your first output is probably improvable but you don't want to fall into infinite polish mode. | [`bounded-self-revision`](../output-quality/bounded-self-revision.md) | Finite revision cycles. After N rounds, stop. Unlike `self-verify-pipeline`, this improves *quality*, not correctness. |
| One correct solution, but you're not sure which of several approaches is the best one. | [`speculative-drafting-verification`](../execution/speculative-drafting-verification/SKILL.md) | Generates multiple candidates in parallel, verifies each against constraints, selects best. |
| Multiple approaches and you want process-reward scoring to pick the winner, not just constraint checking. | [`speculative-exploration-protocol`](../execution/speculative-exploration-protocol.md) | Speculative drafting + ToT + PRM fused. More thorough than `speculative-drafting` but uses more tokens. |
| An important decision — you suspect cognitive bias might be contaminating your reasoning. | [`cognitive-bias-checklist`](../judgment-and-routing/cognitive-bias-checklist.md) | Checks 9 established biases. Use in slow-mode reasoning (important decisions, estimates, architecture choices). |
| Same situation, but you want automated detection — a script that objectively checks for bias patterns. | [`cognitive-bias-auditor`](../judgment-and-routing/cognitive-bias-auditor/SKILL.md) | Companion script detects 9 biases from arXiv:2410.02820. More objective than self-checking the checklist. |
| You generated an answer and want to verify it's not self-contradictory by cross-checking against an independent reasoning path. | [`self-consistency`](../output-quality/self-consistency.md) | Generates multiple independent reasoning paths to the same conclusion. If they diverge, the answer is unreliable. |
| An existing draft/plan/code-change exists and you want a structured critique using tools, not just text review. | [`tool-interactive-critic`](../output-quality/tool-interactive-critic.md) | Multi-tool critique process. Unlike `self-verify-pipeline`, this assumes the output *is final* and you're refining, not catching errors. |
| You need to call an external library but aren't sure of the exact API — you want to avoid hallucinated method signatures. | [`api-surface-anchoring`](../software-development/api-surface-anchoring/SKILL.md) | Verifies current API surface from authoritative docs before writing code. Prevents hallucinated signatures. |
| Same situation, but you want *every* API call to be fully traceable to documentation you actually checked — zero hallucinated APIs, with citations. | [`verified-api-workflow`](../software-development/verified-api-workflow/SKILL.md) | Each verified API surface entry becomes a durable anchor. Code is fully traceable to docs. Use when hallucinated APIs would cause production incidents. |

### Architecture / Design Decisions

| When this is happening… | Use this skill | Why this one (and not the others) |
|---|---|---|

Includes MCP server. Unlike `requirement-crystallization`, this targets *over-engineering*, not unclear requirements. |
| A big decision with multiple valid options — you need to stress-test your choice against alternatives. | [`counterfactual-policy-testing`](../judgment-and-routing/counterfactual-policy-testing.md) | Tests against null/opposite/partial alternatives.
